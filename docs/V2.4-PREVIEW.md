# MACS v2.4 Preview: Enhanced Data Structures

> 这是 v2.4 的概念预览，展示增强的数据结构能带来什么能力

---

## 🎯 核心思想

**v2.3**: Markdown → AST → 结构化数据（扁平）
**v2.4**: Markdown → AST → 结构化数据 → **关系图（图数据库）**

---

## 📊 数据结构对比

### v2.3 (Current)

```typescript
interface ChangelogEntry {
  date: string
  type: 'feat' | 'fix' | ...
  content: string
  author: string
  tags: string[]
  lineRange: [number, number]
  raw: string
}
```

**能做什么**：
- ✅ 按类型/日期/作者/标签查询
- ✅ Token 节省 99%

**不能做什么**：
- ❌ 不知道这个变更关联哪个任务
- ❌ 不知道改了哪些文件
- ❌ 不知道影响程度
- ❌ 不能追溯任务依赖链

---

### v2.4 (Enhanced)

```typescript
interface ChangelogEntryV2 {
  // v2.3 字段（保留）
  date, type, content, author, tags, lineRange, raw

  // 新增：唯一 ID
  id: "C-001"

  // 新增：Git 元数据（从 git log 提取）
  git?: {
    commitHash: "abc123"
    filesChanged: ["src/auth.ts", "src/middleware/jwt.ts"]
    linesAdded: 150
    linesDeleted: 20
    timestamp: "2026-02-28T10:00:00Z"
  }

  // 新增：关系
  references?: {
    tasks: ["T-001"]              // 实现了任务 T-001
    files: ["src/auth.ts"]        // 修改了这些文件
  }

  // 新增：语义标签
  impact: "high"                  // 影响程度（从 lines changed 推断）
  category: "auth"                // 从 tags 聚合
  isBreakingChange: false
}
```

**能做什么（新增）**：
- ✅ 查询"所有与任务 T-001 相关的变更"
- ✅ 查询"所有修改了 src/auth.ts 的变更"
- ✅ 查询"所有高影响变更"
- ✅ 可视化：任务 → 变更 → 文件的关系图

## 🔍 查询示例对比

### v2.3: 基础查询

```typescript
// 只能按基础字段查询
const features = query.queryChangelog({
  type: 'feat',

  author: 'engineer-sonnet'
```

### v2.4: 关系查询

```typescript
// 1. 查找任务 T-001 的所有变更
const taskChanges = query.queryChangelog({
  taskId: 'T-001'
})
// 返回：所有 references.tasks 包含 "T-001" 的变更

// 2. 查找改了 src/auth.ts 的所有变更
const authChanges = query.queryChangelog({
  filesChanged: ['src/auth.ts']

// 返回：所有 git.filesChanged 包含该文件的变更

// 3. 查找高影响变更（改了 100+ 行）
const highImpact = query.queryChangelog({
  impact: 'high'
})

// 4. 追溯依赖链
const blockers = query.getBlockingChain('T-003')
// 返回：T-003 → 被 T-001 阻塞 → T-001 被 T-005 阻塞
// 完整依赖链：[T-005, T-001, T-003]

// 5. 查找文件的变更历史
const fileHistory = query.getFileHistory('src/auth.ts')
// 返回：所有修改过该文件的变更（按时间排序）
```

## 🛠️ 实现方式

### 数据提取

**1. 从 Markdown 提取基础数据**（v2.3 已有）
```typescript
// 已有：解析 CHANGELOG.md
const entry = {
  date: "2026-02-28",
  type: "feat",
  content: "Implement JWT auth #T-001",
  author: "engineer-sonnet",
  tags: ["#dev", "#auth"]
}
```

**2. 增强：提取 taskId**（从 content 中）
```typescript
const taskIdMatch = entry.content.match(/#T-\d+/)
if (taskIdMatch) {
  entry.references = { tasks: [taskIdMatch[0].slice(1)] }  // "T-001"
}
```

**3. 增强：从 Git 提取元数据**
```typescript
const commit = findGitCommit(entry.date, entry.content)
if (commit) {
  entry.git = {
    filesChanged: commit.files,
    linesAdded: commit.stats.insertions,
    linesDeleted: commit.stats.deletions
  }
}
```

**4. 推断影响程度**
```typescript
const totalLines = entry.git.linesAdded + entry.git.linesDeleted
entry.impact =
  totalLines > 100 ? 'high' :
  totalLines > 20 ? 'medium' : 'low'
```

### 关系图构建

```typescript
function buildRelationshipGraph(index: MACSIndexV2): RelationshipGraph {
  const nodes: GraphNode[] = []
  const edges: GraphEdge[] = []

  // 1. 添加任务节点
  index.tasks.forEach(task => {
    nodes.push({
      id: task.id,
      type: 'task',
      label: task.title,
      metadata: task
    })

    // 添加依赖边
    task.dependencies?.blockedBy?.forEach(blockerId => {
      edges.push({
        from: blockerId,
        to: task.id,
        type: 'blocks'
      })
    })
  })

  // 2. 添加变更节点 + 关系边
  index.changelog.forEach(change => {
    nodes.push({
      id: change.id,
      type: 'change',
      label: change.content,
      metadata: change
    })

    // 变更 → 任务
    change.references?.tasks?.forEach(taskId => {
      edges.push({
        from: change.id,
        to: taskId,
        type: 'implements'
      })
    })

    // 变更 → 文件
    change.git?.filesChanged?.forEach(file => {
      edges.push({
        from: change.id,
        to: `file:${file}`,
        type: 'modifies'
      })
    })
  })

  return { nodes, edges }
}
```

## 📊 Dashboard 可视化示例

### 任务依赖图（D3.js Force Graph）

```html
<div id="task-graph"></div>

<script>
const graph = buildTaskDependencyGraph(index)

const simulation = d3.forceSimulation(graph.nodes)
  .force('link', d3.forceLink(graph.edges))
  .force('charge', d3.forceManyBody())
  .force('center', d3.forceCenter(width / 2, height / 2))

// 节点颜色
node.style('fill', d => {
  if (d.status === 'completed') return 'green'
  if (d.status === 'blocked') return 'red'
  return 'blue'
})

// 边类型
link.style('stroke', d => {
  if (d.type === 'blocks') return 'red'
  return 'gray'
})
</script>
```

**效果**：
- 完成的任务显示为绿色
- 阻塞的任务显示为红色
- 阻塞关系用红线连接
- 鼠标悬停显示详情

## 🤔 要不要做？

### 优点

✅ **技术深度**：从文档模板 → 图数据库级别的分析工具
✅ **实际价值**：解决真实痛点（依赖追溯、影响分析）
✅ **护城河**：需要理解图算法 + Git + 数据可视化，难复制
✅ **差异化**：AutoGen/CrewAI 都没有这个

### 缺点

❌ **开发时间**：4周（如果现在做，会推迟发布）
❌ **复杂度**：增加维护成本
❌ **市场需求**：不确定用户是否真的需要（没验证过）

### 我的建议

**分阶段发布**：

1. **v2.3 先发布**（本周）
   - 已有功能足够差异化（99% token 节省 + Dashboard）
   - 快速获得市场反馈

2. **收集反馈**（1-2周）
   - 看用户最想要什么
   - 看哪些功能被频繁问到

3. **v2.4 按需开发**（1个月后）
   - 如果用户要"任务依赖可视化" → 做关系图
   - 如果用户要"更好的搜索" → 做 QMD 集成
   - 如果用户要"手机端" → 做 MyCC 集成

**理由**：
- 避免过度工程（YAGNI: You Ain't Gonna Need It）
- 让市场决定方向
- 快速迭代 > 完美主义
